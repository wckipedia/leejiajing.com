# Ideas

These ideas are intentionally unscheduled. Move an item to `roadmap.md` only after
its user value, scope, and performance cost are understood.

## Project storytelling

- Dedicated case studies for the strongest projects.
- Architecture diagrams that explain real system decisions.
- Short “problem / decision / result” summaries for recruiter scanning.
- A clear statement of personal contribution on team projects.
- Links to selected source, demos, write-ups, or technical artifacts.

## Engineering credibility

- A concise “how I work” section grounded in examples.
- Project quality signals such as testing, accessibility, observability, and deployment.
- A small changelog or “recently built” feed generated from maintained data.
- Optional downloadable one-page project brief for interview preparation.

## Interaction experiments

- A project filter only if the project set grows enough to need one.
- Progressive architecture diagrams with accessible static fallbacks.
- A restrained command palette for keyboard navigation.
- Contextual code or terminal motifs used as content, not decoration.

## Content experiments

- Technical notes or short engineering write-ups.
- A “currently learning” area with a clear update cadence.
- A photography page only if it strengthens the personal narrative.
- Testimonials or recommendations if credible, specific material becomes available.

## Avoid by default

- Skill percentage bars.
- Decorative dashboards with invented metrics.
- Continuous background animation.
- Heavy 3D or WebGL without a narrative and measurable performance budget.
- A blog, CMS, or analytics stack without a real maintenance plan.
